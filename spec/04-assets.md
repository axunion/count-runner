# 04. アセット仕様(画像差し替え計画)

ゲームはプレースホルダー図形で動作するが、世界観の本命は後日制作するキャラクター・オブジェクト・背景の画像である。本章は**「後から制作した画像を、コード変更なしで差し替えられる」**ための規約を先に確定させる。アセット制作者は本章と [01-world-theme.md](./01-world-theme.md) §2(アートディレクション)だけを見て制作できることを目指す。

## 1. 差し替えの仕組み(全体像)

```
制作した画像を規約サイズで書き出す
    ↓
public/assets/themes/fantasy/ に規約名で置く
    ↓
themeConfig.ts にエントリを追記(Canvas 要素は assets、クロームは chrome セクション)
    ↓
pnpm dev で確認 — コンポーネント / render/ / CSS の変更はゼロ
```

- Canvas 描画要素は「アセット定義があり、かつロード成功していれば画像。それ以外はプレースホルダー図形」を要素ごとに自動で選ぶ([03-architecture.md](./03-architecture.md) §5)。
- クローム(ゲーム外周)画像は Canvas を通らず、`chrome` セクションの URL を CSS カスタムプロパティ経由で `background-image` に流す([01](./01-world-theme.md) §3、[03](./03-architecture.md) §9)。未定義なら `chrome.backgroundColor` の 1 色で成立する。
- **部分差し替えが可能**であること: 例えばユニット画像だけ先に完成した場合、ユニットのみ画像・他はプレースホルダーで混在動作する。
- アセットのロード失敗(パス間違い等)はゲームを止めず、プレースホルダーで続行し、コンソールに警告を出す。

## 2. 制作の基本ルール

| 項目 | 規約 |
| --- | --- |
| 解像度基準 | ゲームは論理幅 360px で描画(高さは端末により 640〜900px。[03](./03-architecture.md) §4)。**制作は論理サイズの 2 倍(@2x)のピクセル数**で行う(例: 論理 16×16 の要素 → 32×32px で制作)。PC ではさらに CSS 拡大がかかるが、@2x + 総倍率キャップ 3 の範囲では実用上十分 |
| フォーマット | **PNG(透過)**。背景タイルなど不透過のものも PNG で統一(将来 WebP 化は任意の最適化) |
| アンカー(基準点) | **画像の中心**。ゲームは要素の中心座標に画像の中心を合わせて描く |
| 余白 | 意味のない透明余白を残さない(表示サイズ計算が狂うため)。発光などのはみ出し表現は表示サイズに含めて設計する |
| 向き | キャラクターは**上向き(奥へ走る後ろ姿)**が正位置。ゲームの進行方向は画面上方向であるため |
| 背景タイル | 幅は論理 360px 固定、**上下端がシームレス**に繋がること(縦ループスクロール)。論理高さが可変でもタイルの繰り返しで自然に対応できる |

## 3. 必要アセット一覧(サイズ確定表)

### Canvas 描画アセット(theme.assets)

「ID」は theme.assets のキー名(§4)、「論理表示サイズ」がゲーム内で占める大きさ、「制作サイズ」が実際に書き出す px。ファイル名は §6。優先度: ★★★ = 世界観の体感に最も効く、★☆☆ = 無くても成立。

| ID | 用途 | 論理表示サイズ | 制作サイズ(@2x) | 構成 | 優先度 |
| --- | --- | --- | --- | --- | --- |
| `unit` | 見習い魔法使い(群れの 1 体) | 16×16 | 32×32 × 4 フレーム → **128×32 の横 1 列シート** | 走りアニメ 4 フレーム、8fps 目安 | ★★★ |
| `enemyUnit` | 敵集団(Stone Legion)の 1 体 | 16×16 | 128×32(unit と同じ 4 フレームシート) | guard クラスタで共有。省略時は敵色の円 | ★★☆ |
| `leaderGlyph` | リーダー足元の魔法陣 | 32×32 | 64×64 | 1 枚。**回転はコード側**で行うため画像は静止で描く | ★★☆ |
| `gateAdd` | Mana Font(マナの泉)バナー | 164×56 | 328×112 | 1 枚。通過済みの減光はコード側の透明度で行うため状態違い不要 | ★★★ |
| `gateMultiply` | Chronos Gate(刻の門)バナー | 164×56 | 328×112 | 同上 | ★★★ |
| `gateSubtract` | Gargoyle Wall(敵壁)バナー | 164×56 | 328×112 | 同上 | ★★★ |
| `boss` | Gate Guardian(フィナーレのボス) | 120×120 | 240×240 | 1 枚。HP 数値はコード側が描く | ★★☆ |
| `background` | フィールド背景タイル | 360×320(縦ループ) | 720×640 | 上下シームレス。地面 + 荒野の小物 | ★★☆ |
| `goalBanner` | SANCTUM(ゴールの門) | 360×80 | 720×160 | 1 枚 | ★★☆ |
| `overlayClear` | クリア画面の挿絵 | 240×160 | 480×320 | 1 枚(DOM の `<img>` で表示) | ★☆☆ |
| `overlayGameOver` | ゲームオーバー画面の挿絵 | 240×160 | 480×320 | 同上 | ★☆☆ |

サイズの根拠:

- **unit 16×16**: プレースホルダー円(半径 5 = 直径 10)より一回り大きい。ユニットの中心間隔は最小約 7px([03](./03-architecture.md) の `FORMATION_SPACING`)で意図的に重なり合い、群体として見える。300 体時の群れ直径 ≈ 240px に対して画面幅 360px に収まる。
- **gate バナー 164×56**: レーン幅(左セル x 12〜176)と行高さ `ROW_HEIGHT = 56` にちょうど一致。`displayValue`("+7" 等)の文字はコード側が上描きするため、**画像内に効果数字を焼き込まない**こと(値は毎回抽選されるため焼き込みは成立しない)。名称ロゴ(Mana Font 等)は画像に含めてよい。
- **boss 120×120**: 画面上部で群れ(直径 ≈240px)と対峙して見劣りしない大きさ。HP 数値・`hpLabel` はコード側が描くため焼き込まない。
- **background 360×320**: 縦ループタイル。論理高さが 640〜900 のどれでも「必要枚数 + 1 枚」の繰り返しで隙間なく敷ける。1 枚を大きくしすぎない(初回ロード対策)。

### クローム画像(theme.chrome。CSS 配信)

Canvas に描かないため論理サイズの規約はなく、**推奨サイズ + CSS の cover/repeat 前提**で制作する。

| ID | 用途 | 推奨サイズ | 表示方法 |
| --- | --- | --- | --- |
| `chrome-background` | ページ全面の背景 | 1920×1280 以上 | `background-size: cover`(中央基準で切り抜かれる前提の構図にする) |
| `chrome-side-panel` | ワイド画面の左右装飾列 | 480×1440(縦長) | 列に対して cover。左右で同一画像(ミラー表示はしない) |

- クローム画像は**装飾**であり、切り抜かれても破綻しない構図(端に重要要素を置かない)にする。
- ゲーム領域より一段暗いトーンにする([01](./01-world-theme.md) §2)。

## 4. themeConfig.ts の assets セクション(型仕様)

[01-world-theme.md](./01-world-theme.md) §3 の `ThemeAssetConfig` に含まれる `assets` の定義。

**SpriteAsset(1 エントリの型)**

| フィールド | 型 | 必須 | 意味 |
| --- | --- | --- | --- |
| `src` | string | ✓ | 画像パス(`/assets/themes/fantasy/...`。public 配下なのでルート絶対パス) |
| `displayWidth` / `displayHeight` | number | ✓ | 論理表示サイズ(§3 の表の値) |
| `frameCount` | number | — | シートのフレーム数。省略時 1(静止画) |
| `fps` | number | — | アニメ再生速度。省略時 8 |

**ThemeAssets(assets セクションの型)** — 以下のキーをすべて **optional** で持つ:

`unit` / `enemyUnit` / `leaderGlyph` / `gateAdd` / `gateMultiply` / `gateSubtract` / `boss` / `background` / `goalBanner` / `overlayClear` / `overlayGameOver`(すべて SpriteAsset)

- ゲート 3 種のキーは GateKind と機械的に対応付けられる命名にする(描画分岐で `kind` から引けること)。
- **未定義キー = プレースホルダー描画**。これが差し替えの ON/OFF スイッチであり、他のフラグを設けない。
- クローム画像はこの型に**含めない**(`chrome` セクションの素の URL 文字列。[01](./01-world-theme.md) §3)。

## 5. 描画側の対応規則(実装要件)

| 要素 | 画像があるときの描画 |
| --- | --- |
| unit | 全ユニットが同一シートを共有。フレーム番号は `floor((elapsed × fps + 位相) mod frameCount)` で経過時間から算出(位相は `wobblePhase` を流用し、群れが一斉に同じポーズにならないようにする)。1 体 1 回の drawImage。**回転・per-unit の状態保持・save/restore は禁止**([03](./03-architecture.md) §5) |
| enemyUnit | guard クラスタの各体に unit と同じ規則で描く。未定義時は敵色(`theme.enemy.color`)の円クラスタ |
| leaderGlyph | リーダー座標を中心に `elapsed × GLYPH_ROT_SPEED` の回転をかけて描く(save/translate/rotate/restore はフレーム 1 回のみなので許容)。フィーバー中の発光表現(色・拡大)は画像にも適用する |
| gateAdd / gateMultiply / gateSubtract | セル中心にバナーを描き、その上にコードで `cell.displayValue` 文字を重ねる。`resolved` 時は透明度を落とす |
| boss | `BOSS_Y` を中心にボス画像を描き、その下にコードで `hpLabel` + HP 数値を重ねる |
| background | タイルを `distance mod タイル高さ` のオフセットで縦に並べてループ描画(枚数は `ceil(viewH / タイル高さ) + 1`)。縞・分割線のプレースホルダー描画は行わない |
| goalBanner | ゴールライン位置にバナーを描く。プレースホルダーの線 + ラベルは行わない |
| overlayClear / overlayGameOver | Canvas ではなく DOM オーバーレイ内の画像として表示(定義が無ければ画像なしの文字のみ) |
| chrome-* | Canvas を通らない。`chrome` セクションの URL を CSS 変数として注入([03](./03-architecture.md) §9)。ロード失敗時はブラウザが背景画像を描かないだけで、地色(`chrome.backgroundColor`)が残る(専用のフォールバック処理は不要) |

**プリロード**: ゲーム開始時(マウント時)に assets の定義済みエントリをすべて画像ロードし、ロード完了を要素ごとに記録する(`theme/assetLoader.ts`)。ロード完了までの間および失敗時はプレースホルダーで描画し続ける(ロード待ちでゲームを止めない)。

## 6. ファイル配置と命名

```
public/
  assets/
    themes/
      fantasy/
        unit.png              (128×32, 4 フレーム)
        enemy-unit.png        (128×32, 4 フレーム)
        leader-glyph.png      (64×64)
        gate-add.png          (328×112)
        gate-multiply.png     (328×112)
        gate-subtract.png     (328×112)
        boss.png              (240×240)
        background.png        (720×640)
        goal-banner.png       (720×160)
        overlay-clear.png     (480×320)
        overlay-gameover.png  (480×320)
        chrome-background.png (1920×1280 以上)
        chrome-side-panel.png (480×1440)
```

- ファイル名は assets のキーのケバブケース(例外: `overlayGameOver` → `overlay-gameover.png`。gameover は 1 語扱い)。テーマ追加時は `themes/<テーマ名>/` ディレクトリを増やす。
- 実画像を配置するまでこのディレクトリ自体を作らない(themeConfig の assets も空オブジェクト、chrome も色のみで開始する)。パス規約は本セクションの構成図で確定済み。

## 7. 差し替え検証(受け入れ手順)

実装完了時、以下の手順で差し替え機構が機能することを確認する(本物のアセットはまだ無いため、**ダミー画像**で検証する):

1. 適当な単色 PNG(サイズは §3 の規約通り)を `unit.png` と `gate-add.png` として配置する(ImageMagick 等での生成、あるいは手元の任意画像のリサイズでよい)。
2. `themeConfig.ts` の assets に `unit` と `gateAdd` の 2 エントリを追記する。
3. `pnpm dev` → ユニットと加算ゲートだけが画像になり、**他はプレースホルダーのまま混在動作**することを確認する。
4. `src` をわざと存在しないパスに変えて、警告が出つつプレースホルダーで動き続けることを確認する。
5. 適当な PNG を `chrome-background.png` として配置し、`chrome.backgroundImageSrc` を設定 → ページ背景が画像になることを確認する。エントリを外すと `backgroundColor` の 1 色に戻ることを確認する。
6. すべてのエントリを削除し、全プレースホルダーに戻ることを確認する。**手順全体を通して themeConfig.ts 以外のソースファイルに一切触れていないこと**。
7. 検証後、ダミー画像と追記エントリは削除して初期状態に戻す。
