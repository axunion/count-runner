# 02. ゲームデザイン(面白さの設計)

このゲームが「動く」だけでなく「面白い」ためのデザインを定義する。実装中にプレイフィールの判断に迷ったら、この章を判断基準にする。バランス数値は [03-architecture.md](./03-architecture.md) §10 の定数表と対応している。

## 1. コアファンタジーとコアループ

**コアファンタジー**: 「自分の判断ひとつで、仲間の群れが雪だるま式に増えていく(あるいは失われていく)」。プレイヤーは 1 体の操作ではなく**群れの運命**を握っている。

**コアループ**(1 サイクル約 1.4 秒 = 1 ゲート行):

```
①観察: 次の行の左右ゲートを見る(約 2.5 秒先まで見えている)
    ↓
②判断: どちらが得か? 今の所持数だと ×2 と +5 はどちらが上か?
    ↓
③操作: リーダーをドラッグして左右のレーンへ寄せる
    ↓
④結果: 群れが目に見えて増える / 減る + MANA POWER が跳ねる
    ↓ (すぐ次の行が迫る)
①へ戻る
```

このループが約 21 回連続し、最後にゴール(SANCTUM)での解放が来る。**④の「結果が群れのサイズとして視覚的に返ってくる」こと**がこのジャンルの快感の中核であり、フィードバック演出(§5)は④を強化するために存在する。

## 2. 面白さの 3 本柱

| 柱 | 内容 | 支える実装要素 |
| --- | --- | --- |
| **A. 増殖の快感** | 群れが物理的に画面を埋めていく。数字ではなく「面積」で成長を感じる | ひまわり配置(数に応じて群れの半径が √n で拡大)、×2 での爆発的増加、MAX 300 の圧巻の画 |
| **B. 二択の緊張と手応え** | 毎行が小さなクイズ。「明らかな正解」と「所持数に依存する正解」が混ざる | 行パターン設計(§3)、×2 の動的価値、danger 行の被害最小化 |
| **C. 喪失のリスク** | 仲間が石にされる痛み。0 になったら終わりという崖 | 初期 10 体の少なさ(序盤は -3 が致命傷になり得る)、GameOver の存在、外周から消える減少表現 |

3 本柱のどれかを損なう実装・調整はしない。例: ユニットを数字表示だけにする(A を殺す)、常に片側が正解になる(B を殺す)、0 になっても続行できる(C を殺す)。

## 3. 二択判断の設計

### 行パターンと、それが生む判断

各行は `rollRowPattern`(乱数注入の純関数)で決まる。パターンごとに**プレイヤーの頭の中に起きること**を設計する:

| パターン | 生む判断 | 感情 |
| --- | --- | --- |
| 加算 vs 敵(+5 / -3) | 反射的な正解選び。操作の練習 | 安心。「見て、避けて、取る」の基本形 |
| 乗算 vs 敵(×2 / -3) | 正解は明白だが報酬が大きい | 高揚。「絶対取りたい」→操作が丁寧になる |
| 乗算 vs 加算(×2 / +5) | **所持数依存の最適化**。5 体以下なら +5、6 体以上なら ×2 が得 | 思考。「今何体だっけ?」と HUD を見る瞬間を作る |
| 敵 vs 敵(-3 / -3) | 回避不能。被害を受け入れる | 緊張のスパイク。「守りのフェーズ」 |

**×2 の動的価値**がこのゲームの判断の深さの源泉である。序盤(所持 10 体)では ×2 = +10 だが、終盤(100 体)では ×2 = +100 になる。同じゲートの価値がプレイを通じて変わるため、「乗算 vs 加算」行は毎回考える価値がある。

### 距離フェーズ別の出現率(難易度曲線)

一定の出現率では「序盤が難しく、終盤が単調」になる(序盤は 10 体しかいないので敵が致命的、終盤は増えすぎて敵が無痛)。距離に応じて 3 フェーズで出現率を変える:

| パターン | Phase 1: 学習(0〜2000px) | Phase 2: 最適化(2000〜4200px) | Phase 3: 試練(4200px〜ゴール) |
| --- | --- | --- | --- |
| 加算 vs 敵 | 60% | 35% | 25% |
| 乗算 vs 敵 | 25% | 25% | 20% |
| 乗算 vs 加算 | 15% | 30% | 25% |
| 敵 vs 敵(danger) | 0% | 10% | 30% |

- **Phase 1** は正解が明白な行を中心にし、操作と ルールの学習に充てる。danger 行は出さない(理不尽な初見殺しの排除)。
- **Phase 2** で「乗算 vs 加算」を増やし、所持数を見て考えるプレイに移行させる。
- **Phase 3** は danger 行で蓄えた群れを削り、「大群で駆け抜ける爽快」と「削られる緊張」を同時に作る。ゴール前 400px は行を出さず(GOAL_SAFETY)、最後は群れを眺めながらゴールする「勝利の花道」にする。
- good/bad ゲートの左右配置は毎行 50/50 でロールする(パターン暗記の防止)。
- フェーズ境界と確率は調整ノブ(§6)。`rollRowPattern` はフェーズ(または距離)を引数に取る設計とし、テストで各フェーズの分布を検証できるようにする。

### 期待値の検算(この設計が成立している根拠)

- 総行数 ≈ (6000 − 400) / 260 ≈ **21 行**(Phase 1: 約 8 行 / Phase 2: 約 8 行 / Phase 3: 約 5 行)。
- 常に有利側を選んだ場合: Phase 1 終了時 ≈ 40〜60 体 → Phase 2 で乗算が絡み 150〜300 体 → Phase 3 の danger(-3 数回)は誤差 → **上限 300 付近でゴール**。上手いプレイは「画面が魔法使いで埋まる」報酬を得る。
- 悪い側を引き続けた場合: 初期 10 体は -3 を 3〜4 回で全滅する。**Phase 1 でも操作を放棄すれば負けられる**(緊張が最初からある)。
- 中間(半々で選ぶ)場合: 20〜60 体程度でゴール。クリアはできるが「もっと増やせたはず」が残り、リトライ動機になる。

## 4. テンポとセッション設計

| 項目 | 値 | 意図 |
| --- | --- | --- |
| 1 プレイの長さ | 約 33〜35 秒 | 「もう 1 回」が軽い。失敗のコストが小さい |
| 判断の間隔 | 約 1.4 秒 / 行 | 忙しすぎず、退屈しない。ドラッグ移動は最大幅でも約 0.5 秒で完了する |
| ゲートの視認猶予 | 約 2.5 秒(スポーンから判定線まで 570px ÷ 180px/s) | 「見えていたのに避けられなかった」を起こさない = 失敗の納得感 |
| リトライ | オーバーレイのボタン 1 タップで即再開 | GameOver の悔しさが熱いうちに次のプレイへ |

**失敗の納得感**を支える技術要件(実装は [03-architecture.md](./03-architecture.md)):
- ゲート効果はデータとして事前に画面に表示されている(`displayValue` + `label`)。隠し情報はない。
- dt クランプにより、タブ復帰やフレーム落ちで行が「ワープして当たる」ことがない。
- 判定はリーダー位置基準で一意(左レーンか右レーンか)。曖昧な当たり判定を作らない。

## 5. フィードバック演出(ジュース)

コアループ④「結果」を強化する演出。プロトタイプでも **Must は必須**とする。世界観アセットが未導入の段階でも、演出があるだけで「ゲームらしさ」が大きく変わるため。

| 優先度 | 演出 | 仕様 |
| --- | --- | --- |
| **Must** | 群れサイズの即時変化 | 増加分はゲート位置から湧いて群れに合流(lerp が演出を兼ねる)。減少分は群れの外周から消える |
| **Must** | HUD の数値更新 | ゲート通過の瞬間に MANA POWER が変わる。ユニット数と常に一致 |
| **Must** | 通過済みゲートの減光 | どの行をもう処理したかが一目で分かる |
| **Should** | フローティングテキスト | ゲート通過時、通過セルの位置にテーマの `displayValue`(例 "+5")をゲート色で表示し、上方向へ流れながらフェードアウト(寿命 0.8 秒)。**非リアクティブ**な配列(world 内)で管理し Canvas に描く |
| **Should** | HUD スコアのパンチ | ユニット数が変わった瞬間、`.hudValue` に CSS の scale アニメーション(1.0→1.25→1.0、約 200ms)を再トリガー |
| **Should** | ×2 通過時の強調 | フローティングテキストを他より大きく描く(乗算はジャックポットである) |
| Later(スコープ外) | パーティクル、効果音、被弾時の画面シェイク、石化アニメーション | アセット・演出強化フェーズで実施 |

## 6. 調整ノブ(バランス調整ガイド)

プレイテスト後に「どう感じたら、どの定数を触るか」の対応表。定数の実値は [03-architecture.md](./03-architecture.md) §10。

| 感じた問題 | 触るノブ | 方向 |
| --- | --- | --- |
| 忙しすぎる / 退屈 | `SCROLL_SPEED`、`ROW_INTERVAL` | 速度を下げる / 間隔を広げる |
| 1 プレイが長い・短い | `GOAL_DISTANCE` | 6000px ± 1500px の範囲で |
| 序盤で死にすぎる | `INITIAL_UNITS`、Phase 1 の敵出現率 | 初期数を増やす / danger をさらに遅らせる |
| 終盤に緊張がない | Phase 3 の danger 率、`gates.subtract.value` | danger を増やす(ただし 40% 上限目安) |
| 増殖の快感が弱い | `gates.multiply.value`、`MAX_UNITS` | ×2→×3 は劇薬(インフレ注意)。まず MAX を上げて画の迫力で解決 |
| 操作が重い / 軽すぎる | `LEADER_LERP_RATE` | 8(重い)〜16(ピタ付き)の範囲 |
| 群れがだらしなく見える | `UNIT_FOLLOW_RATE`、`FORMATION_SPACING` | 追従を上げる / 間隔を詰める |

## 7. 面白さのプレイテスト観点(自己検証チェックリスト)

実装完了時、以下を実際にプレイして自問し、結果を報告すること。NO が 2 つ以上あればリリース前に §6 のノブで調整する。

- [ ] 「乗算 vs 加算」の行で、**自分の所持数を確認してから選ぶ瞬間**があったか(B の柱)
- [ ] ×2 を通過した瞬間、群れの膨張に**気持ちよさ**があったか(A の柱)
- [ ] danger 行(敵 vs 敵)が近づいたとき、**身構える感覚**があったか(C の柱)
- [ ] GameOver になったとき、「操作/判断を間違えた自分のせい」と感じられたか(理不尽でないか)
- [ ] Clear 後、「次はもっと増やせる」と**すぐリトライしたくなった**か
- [ ] 30 秒のプレイ中、**待つだけの時間**(判断も操作もない区間)が 3 秒以上続かなかったか(GOAL_SAFETY 区間を除く)
- [ ] 初見のつもりで見て、各ゲートの良し悪しが**色と表記だけで即座に**判別できたか
