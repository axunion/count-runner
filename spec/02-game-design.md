# 02. ゲームデザイン(面白さの設計)

このゲームが「動く」だけでなく「面白い」ためのデザインを定義する。実装中にプレイフィールの判断に迷ったら、この章を判断基準にする。バランス数値は [03-architecture.md](./03-architecture.md) §10 の定数表と対応している。

## 0. v1 の診断(v2 が解決する単調さ)

v1 は成立していたが単調だった。原因は 4 つの欠落に整理できる:

| 欠落 | v1 の症状 | v2 の対策 |
| --- | --- | --- |
| **判断の非自明性** | ゲート値が +5 / ×2 / -3 の固定 3 種のみ。「乗算 vs 加算」以外は正解が即答でき、21 行の大半が作業になる | ゲート値の抽選(§3)、敵集団に守られたゲート(§3) |
| **クライマックス** | ゴールが「線」でしかない。増やした群れを**使う場面**がなく、増殖が自己目的化 | ボス戦フィナーレ(§5) |
| **スキル表現** | 正解側に寄せる以外に上手さが出ない | コンボ/フィーバー(§4)、終盤加速(§7) |
| **リトライ動機** | 結果が「体数」のみ。伸びしろの可視化がない | スコアとベスト記録(§6) |

## 1. コアファンタジーとコアループ

**コアファンタジー**: 「自分の判断ひとつで、仲間の群れが雪だるま式に増えていく(あるいは失われていく)」。プレイヤーは 1 体の操作ではなく**群れの運命**を握っている。v2 はこれに「**育てた群れを最後の総力戦にぶつける**」という結末を加える。

**コアループ**(1 サイクル約 1.1〜1.4 秒 = 1 ゲート行):

```
①観察: 次の行の左右ゲートを見る(値は毎回違う。約 2.5〜4.5 秒先まで見えている)
    ↓
②判断: どちらが得か? 「+7 vs ×2」は今の所持数だと? 敵集団の 40 体を払ってでも ×2 を取るか?
    ↓
③操作: リーダーをドラッグして左右のレーンへ寄せる
    ↓
④結果: 群れが目に見えて増える / 減る + MANA POWER が跳ねる + 正解ならコンボが伸びる
    ↓ (すぐ次の行が迫る。終盤ほど速い)
①へ戻る
```

このループが約 21 回連続し、最後に**ボス戦フィナーレ**(§5)で群れを消費して勝敗が決まり、スコア(§6)が確定する。**④の「結果が群れのサイズとして視覚的に返ってくる」こと**がこのジャンルの快感の中核であり、フィードバック演出(§8)は④を強化するために存在する。

## 2. 面白さの 5 本柱

| 柱 | 内容 | 支える実装要素 |
| --- | --- | --- |
| **A. 増殖の快感** | 群れが物理的に画面を埋めていく。数字ではなく「面積」で成長を感じる | ひまわり配置(数に応じて群れの半径が √n で拡大)、×2/×3 での爆発的増加、MAX 300 の圧巻の画 |
| **B. 判断の緊張と手応え** | 毎行が小さな計算問題。値が毎回変わるため「明らかな正解」と「所持数依存の正解」が混ざり続ける | ゲート値抽選(§3)、守られたゲート(§3)、コンボによる正誤フィードバック(§4) |
| **C. 喪失のリスク** | 仲間が石にされる痛み。0 になったら終わりという崖 | 初期 10 体の少なさ、danger 行、終盤加速(§7)、外周から消える減少表現 |
| **D. クライマックスと消費** | 増やした群れを最後に使い切って勝つ。増殖が目的から手段になる | ボス戦フィナーレ(§5)。残存体数 = 勝利マージン |
| **E. 記録と上達の可視化** | 「今日の自分 vs 過去の自分」。クリアの先の目標 | スコア + ベスト記録 + NEW RECORD 演出(§6) |

柱のどれかを損なう実装・調整はしない。例: ユニットを数字表示だけにする(A を殺す)、常に片側が正解になる(B を殺す)、0 になっても続行できる(C を殺す)、ボスを必ず勝てる消化試合にする(D を殺す)。

## 3. 二択判断の設計

### ゲート値の抽選(v2 の核)

各セルの効果値は**スポーン時に抽選**する(純関数 `rollGateValue(kind, distance, rng)`。[03](./03-architecture.md) §8)。抽選レンジはフェーズで広がる:

| kind | Phase 1 | Phase 2 | Phase 3 |
| --- | --- | --- | --- |
| add | +4〜+6 | +3〜+8 | +3〜+10 |
| multiply | ×2 のみ | ×2(90%)/ ×3(10%) | ×2(85%)/ ×3(15%) |
| subtract | -2〜-3 | -2〜-6 | -3〜-8 |

- 整数一様抽選(multiply のみ確率ロール)。抽選値はセル生成時に `value` と `displayValue`(theme の `displayPrefix` + 値)として**セルにデータ化**され、以後は表示も判定も一致し続ける([01](./01-world-theme.md) §3)。
- これにより「+8 vs ×2」(所持 8 体が損益分岐)、「-2 vs -7」(danger 行にも良し悪しが生まれる)など、**毎行が固有の計算問題**になる。v1 の「×2 の動的価値」を全ゲートに拡張するものである。
- Phase 1 のレンジを狭くするのは学習のため(値のブレより先にルールを覚えさせる)。

### 行パターンと、それが生む判断

各行のパターンは `rollRowPattern(distance, rng)`(乱数注入の純関数)で決まる。パターンごとに**プレイヤーの頭の中に起きること**を設計する:

| パターン | 生む判断 | 感情 |
| --- | --- | --- |
| 加算 vs 敵 | 反射的な正解選び。操作の練習 | 安心。「見て、避けて、取る」の基本形 |
| 乗算 vs 敵 | 正解は明白だが報酬が大きい | 高揚。「絶対取りたい」→操作が丁寧になる |
| 乗算 vs 加算 | **所持数依存の最適化**。抽選値により損益分岐が毎回動く | 思考。「今何体だっけ?」と HUD を見る瞬間を作る |
| 敵 vs 敵(danger) | 回避不能。ただし値が違うので**マシな方を選ぶ**判断は残る | 緊張のスパイク。「守りのフェーズ」 |
| **守られた乗算 vs 加算(guarded)** | 乗算ゲートの手前に敵集団(Stone Legion)。**通行料を払ってジャックポットを取るか、無料の加算で逃げるか** | 駆け引き。「40 体払って ×2……今 100 体だから払う!」 |

**守られたゲート(guarded)の仕様**:

- 乗算セルの手前に敵集団が陣取る。敵集団の体数はスポーン時に `round(現在の所持数 × rng∈[GUARD_RATIO_MIN, GUARD_RATIO_MAX])`(下限 `GUARD_MIN`)で決まり、**体数バッジとして常時表示**される(隠し情報なし)。
- そのレーンを選ぶと「敵集団の体数を所持数から差し引く → 生き残っていれば乗算を適用」の順で解決する。所持数 ≤ 敵集団数なら全滅 = GameOver。
- スポーンから判定までの数秒で所持数が変わる(直前の行で ×2 を取った等)ため、**表示された通行料の価値がリアルタイムに変動する**のがこのパターンの深さ。
- 例: 所持 100、左 = 40 体の敵集団 + ×2、右 = +6 → 左は (100−40)×2 = 120、右は 106。左が正解。所持 50 なら左 20 vs 右 56 で右が正解。

### 距離フェーズ別の出現率(難易度曲線)

| パターン | Phase 1: 学習(0〜2000px) | Phase 2: 最適化(2000〜4200px) | Phase 3: 試練(4200px〜ゴール) |
| --- | --- | --- | --- |
| 加算 vs 敵 | 60% | 30% | 20% |
| 乗算 vs 敵 | 25% | 20% | 15% |
| 乗算 vs 加算 | 15% | 30% | 25% |
| 敵 vs 敵(danger) | 0% | 10% | 20% |
| 守られた乗算 vs 加算(guarded) | 0% | 10% | 20% |

- **Phase 1** は正解が明白な行を中心にし、操作とルールの学習に充てる。danger / guarded は出さない(理不尽な初見殺しの排除)。
- **Phase 2** で「乗算 vs 加算」と guarded を導入し、所持数を見て考えるプレイに移行させる。
- **Phase 3** は danger と guarded で蓄えた群れを削り、「大群で駆け抜ける爽快」と「削られる緊張」を同時に作る。ゴール前 400px は行を出さず(GOAL_SAFETY)、ボス戦前の「静けさ」にする。
- good/bad ゲートの左右配置は毎行 50/50 でロールする(パターン暗記の防止)。
- フェーズ境界と確率は調整ノブ(§9)。`rollRowPattern` はテストで各フェーズの分布を検証できるようにする([05](./05-implementation-plan.md) §4)。

### 期待値の検算(この設計が成立している根拠)

- 総行数 ≈ (6000 − 400) / 260 ≈ **21 行**(Phase 1: 約 8 行 / Phase 2: 約 8 行 / Phase 3: 約 5 行)。
- 常に最善を選んだ場合: Phase 1 終了時 ≈ 50〜80 体 → Phase 2 で乗算(時々 ×3)が絡み 250〜300 体 → Phase 3 の削りを吸収して **250〜300 体でボス戦へ**。ボス HP 100 を大差で削り切り、勝利マージン 150〜200。
- 悪い側を引き続けた場合: 初期 10 体は -2〜-3 を 4〜5 回で全滅する。**Phase 1 でも操作を放棄すれば負けられる**(緊張が最初からある)。guarded で所持数以上の通行料を払えば即全滅。
- 中間(半々で選ぶ)場合: 50〜120 体でボス戦へ。ボス HP 100 との**僅差の削り合い**になり、勝っても負けても「もっと増やせたはず」がスコアの伸びしろとして可視化される(§6)。

## 4. コンボとフィーバー(正しい判断への報酬)

- 行の解決時、**最善側を選んだか**を純関数 `betterSide(row, count)` で判定する(左右それぞれ解決した場合の結果体数を比較。guarded は通行料込みで計算)。
- 最善側を選ぶとコンボ +1。両側同値の行は**維持**(加算しない)。最善でない側はコンボ 0 にリセット。danger 行でも「マシな方」を選べばコンボが伸びる。
- コンボが `COMBO_FEVER_THRESHOLD`(5)の倍数に到達するたび**フィーバー**が発動: `FEVER_DURATION`(4 秒)の間、加算・乗算ゲートの**増加量が 1.5 倍**(`FEVER_GAIN_MULT`。増加量 = 適用後 − 適用前を 1.5 倍して切り上げ、MAX_UNITS 上限は維持。subtract と guarded の通行料には適用しない)。
- フィーバー中はリーダーグリフが `player.feverColor` で発光し、HUD のコンボ表示(`hud.comboLabel` + 数値)が強調される。コンボが途切れてもフィーバーは残り時間まで継続する。
- 最大コンボ(`maxCombo`)を記録し、スコア(§6)に加算する。

**設計意図**: 値抽選(§3)で判断が難しくなったぶん、「読み勝てているか」の正誤フィードバックと報酬を明示する。フィーバーの増加ブーストは柱 A の快感スパイクを人為的に作る装置でもある。コンボの判定は結果論(選んだ後に分かる)なので、初心者を妨げず上級者の目標になる。

## 5. ボス戦フィナーレ(クライマックス)

`distance ≥ GOAL_DISTANCE` で即クリアではなく、`gamePhase` が `"finale"` に遷移する:

1. スクロールが停止し、画面上部にボス(Gate Guardian)が出現する。ボス HP は `BOSS_HP_BASE`(100)で、**HP 数値が常時表示**される(`boss.hpLabel`)。
2. 自軍ユニットが毎秒 `BATTLE_DRAIN_RATE`(40 体/s)ずつボスへ突撃し、**1:1 で相殺**される(純関数 `applyBattleTick`。[03](./03-architecture.md) §7)。演出は「末尾ユニットがボスへ吸い込まれて消える + ボス HP が同数減る」。
3. **ボス HP が先に 0 → Clear**(残存体数 = 勝利マージンとしてリザルトに表示)。**自軍が先に 0(同時 0 を含む)→ GameOver**。
4. フィナーレ中はポインター入力を無視する(操作フェーズは終わっている。見せ場に集中させる)。フィーバーの効果もフィナーレには適用しない。

**設計意図**: 増やした群れを最後に**使う**ことで増殖に目的が生まれる(柱 D)。300 体が滝のように流れ込んでボスを削る画は最大の見せ場であり、ギリギリ勝ちと大差勝ちの差がそのままスコアになる。ボス HP は「中間プレイでも僅差で勝てるが、雑なプレイは敗北し得る」水準に置く(重要ノブ。§9)。

## 6. スコアとベスト記録(リトライ動機)

- **Clear 時のみ**スコアを確定する: `score = 残存体数 × SCORE_PER_UNIT(10) + 最大コンボ × SCORE_PER_COMBO(30)`(純関数 `computeScore`)。GameOver はスコアなし。
- ベストスコアを localStorage(キー `"count-runner:best"`)に保存する。読み書きは try/catch で保護し、失敗してもゲームは止めない(プライベートブラウジング対策)。
- リザルトオーバーレイに「残存体数(`resultLabel`)/ スコア内訳(`scoreLabel`)/ ベスト(`bestLabel`)」を表示し、更新時は `newRecordLabel` バッジを出す。
- **設計意図**: 「クリアした/しない」の 2 値を「今日の自分 vs 過去の自分」の連続値目標に変える(柱 E)。1 プレイ約 35 秒 + 即リトライの設計と最も相性が良い。

## 7. テンポとセッション設計

| 項目 | 値 | 意図 |
| --- | --- | --- |
| 1 プレイの長さ | 約 30〜35 秒(道中 ~29 秒 + ボス戦 ~1〜3 秒) | 「もう 1 回」が軽い。失敗のコストが小さい |
| スクロール速度 | 180 → 230 px/s へ距離線形で加速(`scrollSpeedAt(distance)`) | 終盤の時間圧 = 緊張の演出。判断間隔は約 1.4 秒 → 約 1.1 秒へ詰まる |
| ゲートの視認猶予 | 最短約 2.5 秒(最高速時・最小画面高)〜 約 4.6 秒(序盤・縦長画面) | 「見えていたのに避けられなかった」を起こさない。**設計制約: 最短でも 2.0 秒を下回らない**こと |
| リトライ | オーバーレイのボタン 1 タップで即再開 | GameOver の悔しさが熱いうちに次のプレイへ |

**失敗の納得感**を支える技術要件(実装は [03-architecture.md](./03-architecture.md)):
- ゲート効果・敵集団の体数・ボス HP はすべてデータとして事前に画面表示されている。隠し情報はない。
- dt クランプにより、タブ復帰やフレーム落ちで行が「ワープして当たる」ことがない。
- 判定はリーダー位置基準で一意(左レーンか右レーンか)。曖昧な当たり判定を作らない。
- 画面が縦に長い端末は視認猶予が伸びる(先読みが楽になる)が、ソロゲームのため公平性の問題とはしない([03](./03-architecture.md) §4)。

## 8. フィードバック演出(ジュース)

コアループ④「結果」を強化する演出。**Must は必須**とする。

| 優先度 | 演出 | 仕様 |
| --- | --- | --- |
| **Must** | 群れサイズの即時変化 | 増加分はゲート位置から湧いて群れに合流(lerp が演出を兼ねる)。減少分は群れの外周から消える |
| **Must** | HUD の数値更新 | ゲート通過の瞬間に MANA POWER が変わる。ユニット数と常に一致 |
| **Must** | 通過済みゲートの減光 | どの行をもう処理したかが一目で分かる |
| **Must** | ボス HP の削り表示 | フィナーレ中、ボス HP 数値がリアルタイムに減る。突撃ユニットはボスへ吸い込まれて消える |
| **Should** | フローティングテキスト | ゲート通過時、通過セルの位置にセルの `displayValue`(例 "+7")をゲート色で表示し、上方向へ流れながらフェードアウト(寿命 0.8 秒)。**非リアクティブ**な配列(world 内)で管理し Canvas に描く |
| **Should** | HUD スコアのパンチ | ユニット数が変わった瞬間、`.hudValue` に CSS の scale アニメーション(1.0→1.25→1.0、約 200ms)を再トリガー |
| **Should** | 乗算通過時の強調 | フローティングテキストを他より大きく描く(乗算はジャックポットである) |
| **Should** | フィーバー演出 | リーダーグリフを `feverColor` で発光(線色の切り替え + 半径をわずかに拡大)。HUD のコンボ数値を強調 |
| **Should** | コンボリセットの明示 | コンボが切れた瞬間、HUD のコンボ表示を一瞬沈める(色を落とす等)。「読み損ねた」ことが伝わる |
| Later(スコープ外) | パーティクル、効果音、被弾時の画面シェイク、石化アニメーション、ボスの登場カットイン | アセット・演出強化フェーズで実施 |

## 9. 調整ノブ(バランス調整ガイド)

プレイテスト後に「どう感じたら、どの定数を触るか」の対応表。定数の実値は [03-architecture.md](./03-architecture.md) §10。

| 感じた問題 | 触るノブ | 方向 |
| --- | --- | --- |
| 忙しすぎる / 退屈 | `SCROLL_SPEED_BASE` / `SCROLL_SPEED_MAX`、`ROW_INTERVAL` | 速度を下げる / 間隔を広げる。視認猶予 2.0 秒の下限は死守 |
| 1 プレイが長い・短い | `GOAL_DISTANCE` | 6000px ± 1500px の範囲で |
| 序盤で死にすぎる | `INITIAL_UNITS`、Phase 1 の敵出現率・subtract レンジ | 初期数を増やす / レンジを狭める |
| 終盤に緊張がない | Phase 3 の danger / guarded 率、subtract レンジ上限 | 増やす(danger + guarded 合計 50% 上限目安) |
| 判断が簡単すぎる / 難しすぎる | 値レンジ(`GATE_ADD_RANGES` 等)、`GATE_X3_RATES` | レンジを広げると分岐点の計算が増える。Phase 1 は狭いまま維持 |
| guarded を誰も選ばない / 選び得 | `GUARD_RATIO_MIN/MAX` | 比率を下げると「払い得」、上げると「罠」に寄る。0.3〜0.6 が駆け引き圏 |
| ボス戦があっけない / 理不尽 | `BOSS_HP_BASE`、`BATTLE_DRAIN_RATE` | HP は中間プレイの期待所持数帯(50〜120)の上限寄り。drain は 1〜3 秒で決着する速度 |
| フィーバーが空気 / 強すぎ | `COMBO_FEVER_THRESHOLD`、`FEVER_DURATION`、`FEVER_GAIN_MULT` | 発動しやすさと報酬のバランス。1.5 倍 → 2 倍は劇薬 |
| 増殖の快感が弱い | `GATE_X3_RATES`、`MAX_UNITS` | ×3 の頻度を上げる(インフレ注意)。まず MAX を上げて画の迫力で解決 |
| 操作が重い / 軽すぎる | `LEADER_LERP_RATE` | 8(重い)〜16(ピタ付き)の範囲 |
| 群れがだらしなく見える | `UNIT_FOLLOW_RATE`、`FORMATION_SPACING` | 追従を上げる / 間隔を詰める |

## 10. 面白さのプレイテスト観点(自己検証チェックリスト)

実装完了時、以下を実際にプレイして自問し、結果を報告すること。NO が 3 つ以上あればリリース前に §9 のノブで調整する。

- [ ] ゲートの**値を読んで比較する瞬間**が毎行あったか(B の柱。値抽選が機能しているか)

## 11. v3 候補(今回見送るもの・見送り理由つき)

| 候補 | 内容 | 見送り理由 |
| --- | --- | --- |
| **Obstacles(回避障害物)** | 行間に静的な柱/壁を置き、接触したユニットだけ死ぬ。ドラッグ精度のスキル表現としては最強 | 唯一 per-unit 衝突が必要。「末尾切り捨て = 最外周から消える」規律と formation 再割当ての設計を壊し、群れ形状は emergent なので「尻尾が当たった」理不尽感も出やすい。実装 L |
| **Dynamic Gates(変動値ゲート)** | セル値がスポーン後に周期振動し、通過時点の値が適用される | スクロール速度をプレイヤーが操作できないため timing skill にならない。fun-per-cost が低い |
| **複数ステージ / エンドレス** | ボス HP・値レンジ・フェーズ境界のパラメータセット化(`StageConfig[]`) | まず 1 ステージの密度を上げるのが先。v2 の定数は将来 `StageConfig` に括り出せる形(名前付き定数の集約)を維持しておく |
