# 05. 実装計画

[01](./01-world-theme.md)〜[04](./04-assets.md) の仕様を、動作確認可能な 10 ステップに分けて実装する。各ステップの検証をパスしてから次へ進むこと。

## 1. 実装上の制約(全ステップ共通・必読)

このリポジトリの設定に由来する制約。違反すると `pnpm check` が落ちる。

1. **`enum` / `namespace` / class parameter properties 禁止**(`erasableSyntaxOnly`)。種別はすべて文字列リテラルユニオン + interface + const object で表現する。
2. **型 import は `import type`**(`verbatimModuleSyntax`)。同一モジュールから値と型の両方を使う場合は import を分けるか type 修飾子を付ける。
3. **相対 import は拡張子付き**(既存流儀: `src/index.tsx` が `"./App.tsx"` を import している)。
4. **Biome**: 文字列はダブルクオート。未使用の変数・引数・定数を残さない。コミット前に `pnpm fix` で自動整形できる。
5. **per-unit リアクティブ禁止**(パフォーマンス要件)。ユニットを `<For>` で DOM 化しない、ユニット配列を signal / store 化しない、ユニットの座標更新でシグナル setter を呼ばない。ユニットに触れる Solid API は**ゼロ**であること。
6. **テーマ・見た目のハードコード禁止**。テーマ由来のカラーコード・名称文字列・画像パスを `GamePrototype.tsx` / `.module.css` に書かない(検証は [01](./01-world-theme.md) §5)。
7. **SolidJS である**。React の `useState` / `useEffect` / `useRef` / `className` は存在しない。シグナルの読み出しは関数呼び出し形式である点に注意。
8. **描画分岐の徹底**([03](./03-architecture.md) §4)。要素の描画箇所では必ず「アセットあり→画像 / なし→プレースホルダー」のヘルパーを通し、ループ本体に描画スタイルを直書きしない。

## 2. 実装ステップ

各ステップは「実装内容 → 検証」の形式。検証には `pnpm dev` での目視確認を含む。

### Step 1: themeConfig.ts ✅ 実装済み

[01-world-theme.md](./01-world-theme.md) §3 の型仕様(assets の型は [04](./04-assets.md) §4)と fantasy テーマの具体値を実装する。assets は空で開始。

- **検証**: `pnpm check` が成功する。

### Step 2: 静的シェル(コンテナ + Canvas + マウント) ✅ 実装済み

CSS Modules(`.root` / `.frame` / `.canvas`)と、canvas を持つコンポーネントの骨格を作る。マウント時に DPR セットアップとテーマ地色の全面塗りを 1 回行う。`App.tsx` を `<GamePrototype />` のマウントのみに書き換える(既存の `App.css` import は外す)。`public/assets/themes/fantasy/` はディレクトリごと作らない([04-assets.md](./04-assets.md) §6 を参照。実画像を配置する際に作成する)。

- **検証**: `pnpm dev` → ブラウザ中央に 360×640 の暗色キャンバスが表示される。DevTools のデバイスモード(モバイル幅)でも中央に収まる。

### Step 3: ゲームループ + スクロール背景 + リーダーグリフ ✅ 実装済み

WorldState の骨格(distance / elapsed / leaderX / targetX / pointerActive)とファクトリ、rAF ループ(dt クランプ・onCleanup)、背景(縞 + 中央分割線)とリーダーグリフのプレースホルダー描画。背景とグリフの描画はこの時点から**描画分岐ヘルパー経由**にしておく(アセット未定義なのでプレースホルダー側が動く)。

- **検証**: 縞が下方向へ一定速度で流れ、画面下部(LEADER_Y)で魔法陣グリフが回転している。タブを切り替えて戻っても表示が飛ばない(dt クランプ)。

### Step 4: ポインター入力 ✅ 実装済み

pointer capture 付きの入力ハンドラ、論理座標変換、クランプ、リーダーの指数 lerp 追従([03](./03-architecture.md) §6)。`.canvas` に `touch-action: none` があることを確認。

- **検証**: ドラッグでリーダーが滑らかに追従する。ポインタをキャンバス外へ出しても追従が続く。離すとその場に留まる。DevTools のタッチエミュレーションでも動く。

### Step 5: 群衆 ✅ 実装済み

Unit 配列、`formationOffset`(named export)、追従 + wobble、ユニットのバッチ描画(プレースホルダー: 単一パス単一 fill)。初期 `INITIAL_UNITS` 体を配置。

- **検証**: 群れがリーダーに遅れて付いてくる(尾を引く)。`INITIAL_UNITS` を一時的に **250** にして DevTools Performance で 60fps(≦16.7ms/frame)を確認し、確認後 10 に戻す。

### Step 6: 行スポーン + ゲート描画 ✅ 実装済み

GateRow、`rollRowPattern(distance, rng)`(named export・フェーズ別出現率は [02](./02-game-design.md) §3)、距離ベース生成、行のスクロールと画面外除去、ゲートセルのプレースホルダー描画(角丸矩形 + displayValue + label)。ゲート描画も描画分岐ヘルパー経由。

- **検証**: 260px 間隔で 2 択のゲート行が流れてくる。色・表記が theme と一致(emerald +5 / purple x2 / crimson -3)。序盤(Phase 1)に敵 vs 敵の行が出ない。

### Step 7: 衝突 + HUD ✅ 実装済み

`applyGate`(named export)、判定線 + resolved フラグ、ユニット増減(ゲートセル位置からの湧き・末尾切り捨て・フォーメーション再割当て)、`unitCount` シグナル、HUD(scoreLabel + 数値 + 進捗バー、CSS 変数注入)。

- **検証**: +5 で群れが増え、×2 で倍になり、-3 で外周から減る。HUD の MANA POWER がユニット数と常に一致する。resolved 行が減光する。×2 を連取しても 300 で頭打ちになり 60fps を維持する。

### Step 8: フィードバック演出 ✅ 実装済み

フローティングテキスト(FloatText 配列・寿命・上昇・フェード、×2 は大きく)、HUD 数値のパンチアニメーション(変化ごとに再トリガー)。[02-game-design.md](./02-game-design.md) §5 の Must / Should をすべて実装。

- **検証**: ゲート通過のたびに "+5" 等がゲート色で浮かび上がって消える。HUD の数値が跳ねる。演出があっても 60fps を維持する。

### Step 9: 勝敗 + オーバーレイ + リトライ ✅ 実装済み

ゴールライン描画(分岐ヘルパー経由)、勝敗判定、進捗 %、Clear / GameOver オーバーレイ(`<Show>`、overlay-* アセット定義があれば挿絵表示)、Retry(WorldState 再生成 + シグナルリセット)。

- **検証**: 完走 → clearTitle のオーバーレイ。subtract を引き続けて 0 → gameOverTitle のオーバーレイ。Retry → 初期状態(10 体・距離 0・0%)から再開。オーバーレイ表示中はドラッグしてもリーダーが動かない。

### Step 10: アセット差し替え検証 + テスト + 仕上げ ✅ 実装済み

1. アセットプリロードと画像描画パス([04](./04-assets.md) §5)が未実装ならここで完成させ、[04-assets.md](./04-assets.md) §7 のダミー画像手順(1〜6)を実施する。
2. `GamePrototype.test.ts` を作成し §4 のテスト仕様を実装する。
3. テーマハードコード検査([01](./01-world-theme.md) §5 の grep 2 種)を実行する。
4. [02-game-design.md](./02-game-design.md) §7 のプレイテスト観点を通しでプレイして自己評価する。

- **検証**: `pnpm test` / `pnpm check` / `pnpm build` がすべて成功する。差し替え検証で `GamePrototype.tsx` / `.module.css` に触れていない。

## 3. 実装順の意図

Step 1〜2 で「見える土台」、3〜5 で「触って気持ちいい」(操作と群れ)、6〜7 で「ゲームになる」(判断と増減)、8 で「面白く感じる」(フィードバック)、9 で「ゲームとして閉じる」(勝敗)、10 で「将来に開く」(差し替え + 品質)。**Step 5 の時点で操作の手触りに違和感があれば、先へ進まず [02](./02-game-design.md) §6 のノブで調整する**こと(後工程ほど手触りの修正コストが上がる)。

## 4. テスト仕様(GamePrototype.test.ts)

純関数 3 つを対象とする。Canvas・DOM・シグナルのテストは行わない(目視検証でカバー)。

| 対象 | ケース | 期待 |
| --- | --- | --- |
| `applyGate` | add, 5, 10 | 15 |
| | multiply, 2, 10 | 20 |
| | subtract, 3, 10 | 7 |
| | subtract, 3, 2 | 0(下限で止まる。負にならない) |
| | add, 5, 298 / multiply, 2, 200 | いずれも 300(MAX_UNITS 頭打ち) |
| `formationOffset` | i = 0 | 原点 (0, 0) |
| | i = 1〜50 の全ペア | 座標が互いに一致しない(重なり配置がない) |
| | 任意の i ≥ 1 | 原点からの距離が「SPACING×√i を縦 0.75 倍した楕円上」に一致(x²+(y/0.75)² = (7√i)²、誤差許容) |
| `rollRowPattern` | Phase 1 の距離 × 固定 rng 全域(0, 0.25, 0.5, 0.75, 0.99 等) | 返るペアが Phase 1 の許容集合のみ。**danger(subtract/subtract)が出ない** |
| | Phase 2 / Phase 3 の距離 × rng 掃引(0〜1 を 0.01 刻み) | 各フェーズの出現率表([02](./02-game-design.md) §3)の境界通りにパターンが切り替わる |
| | 同一入力を 2 回 | 同一出力(rng 注入で決定的であること) |

## 5. 受け入れ基準チェックリスト(最終確認)

実装完了時に以下をすべて確認し、結果を報告すること。

**ユーザー指定の基準(オリジナル 7 項目)**

- [x] **即プレイ可能**: `pnpm dev` 直後にゲームが開始され操作できる(ビルド・ランタイムエラーなし)。→ 10 秒プレイしてコンソールにエラーがないこと
- [x] **リーダー追従と群れ**: リーダーがポインターに滑らかな lerp で水平追従し、群れが遅れて付いてくる。→ 左右に素早くドラッグして確認
- [x] **正/乗算ゲートの効果**: 通過でユニット数と MANA POWER 表示が目に見えて変化する。→ +5 で HUD +5、×2 で HUD 2 倍、群れの面積も増える
- [x] **敵の効果**: Gargoyle Wall(-3)でカウントが減る。→ HUD -3、群れの外周から消える
- [x] **勝敗**: カウント 0 で GameOver、ゴール到達で Clear のオーバーレイ。→ 両方を実際に出し、Retry で完全リセットされること
- [x] **テーマ分離**: コンポーネントロジックにテーマ文字列・色のハードコードがゼロ。→ [01](./01-world-theme.md) §5 の grep 2 種がヒット 0 件
- [x] **性能**: 200+ ユニットを per-unit リアクティブ更新なしで約 60fps。→ INITIAL_UNITS=250 の一時変更 + Performance 計測で 16.7ms/frame 以下(計測後に戻す)。ユニットに触れる Solid API が存在しないことをコードで確認

**本計画書で追加した基準**

- [x] **アセット差し替え可能性**: [04-assets.md](./04-assets.md) §7 のダミー画像検証(部分差し替え・混在動作・失敗時フォールバック)がすべて通り、手順を通して `GamePrototype.tsx` / `.module.css` に触れていない
- [x] **フィードバック演出**: [02](./02-game-design.md) §5 の Must / Should がすべて動作している(フローティングテキスト、HUD パンチ、減光)
- [x] **面白さの自己検証**: [02](./02-game-design.md) §7 のチェックリストを通しプレイで評価し、結果(YES/NO と所感)を報告している。NO が 2 つ以上なら調整ノブで調整してから完了とする

**品質ゲート**
